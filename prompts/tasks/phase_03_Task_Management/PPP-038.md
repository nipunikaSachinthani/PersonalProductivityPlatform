## 2. Objective

> Write Phase 03 light tests: CreateTask validator unit test, GetTasks handler test with pagination, CRUD integration test, and TaskList component test.

This task provides the safety net for task management. The validator unit test confirms all field validations (empty title rejected, valid title accepted, invalid priority rejected). The handler test with mocked `AppDbContext`/in-memory database verifies pagination logic, filtering by status, and ownership filtering. The integration test goes through the full CRUD cycle via WebApplicationFactory — create, read, update, complete, delete — asserting correct status codes and response shapes. The component test renders the TaskList with mocked TanStack Query data, verifies columns render, and tests the empty state message appears when there are no tasks.

## 4. Technical Guidance

**Conventions for this module:**
- Unit test: `CreateTaskCommandValidatorTests` — test each validation rule (empty title → error, 201-char title → error, valid → no error, invalid priority → error, past due date → error)
- Handler test: `GetTasksQueryHandlerTests` — use EF Core InMemory provider or mock `DbSet`, seed test tasks, verify pagination returns correct page, status filter works, sort order correct, only current user's tasks returned
- Integration test: `TasksControllerTests` using `CustomWebApplicationFactory` — POST create → 201, GET list → 200 with array, GET by id → 200, PUT update → 200, POST complete → 200 with Done status, DELETE → 204, GET deleted → 404
- Component test: `TaskList.test.tsx` using Vitest + RTL — mock TanStack Query `useQuery` hook, verify table renders rows, verify empty state message, verify status badges display

**Concepts practiced in this task:**
- Validator unit testing
- Handler testing with in-memory database
- Full CRUD integration testing
- Component testing with mocked data

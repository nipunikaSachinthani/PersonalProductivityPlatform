## 2. Objective

> Implement GetTasks list query with pagination, filtering (status, priority), and sorting (due date, created date) via `GET /api/v1/tasks`.

The GetTasks endpoint powers the task list view — the most-visited page in the app. It supports pagination (`page`/`pageSize` with defaults), filtering by status and/or priority, and sorting by due date or creation date (ascending or descending). The handler uses EF Core's `IQueryable` to build the query dynamically, applies `Skip`/`Take` for pagination, and returns a paginated response envelope with `totalCount` metadata. The authorization rule enforces that users only see their own tasks (filtering by `CreatedByUserId`). This task teaches query composition, pagination patterns, and read-side CQRS.

## 4. Technical Guidance

**Conventions for this module:**
- Query: `GetTasksQuery` in `Application/Tasks/Queries/` with `Page (default 1)`, `PageSize (default 20, max 100)`, `Status?`, `Priority?`, `SortBy? (dueDate/createdAt)`, `SortOrder? (asc/desc)`
- Handler: build `IQueryable<TaskItem>`, filter by `CreatedByUserId == currentUserId`, apply optional status/priority filters, apply sort, apply `Skip((page-1)*pageSize).Take(pageSize)`, execute with `ToListAsync()`
- Also run a `CountAsync()` for total count (separate query to avoid loading all rows)
- Response: `ApiResponse<List<TaskDto>>` with `meta: { page, pageSize, totalCount, totalPages }`
- Endpoint: `TasksController.GetTasks()` → `GET /api/v1/tasks?page=1&pageSize=20&status=Todo&sortBy=dueDate&sortOrder=asc`
- Projection: use `.Select()` or AutoMapper to map to `TaskDto` (avoids loading unnecessary columns)

**Concepts practiced in this task:**
- Query composition with EF Core IQueryable
- Pagination pattern (Skip/Take + total count)
- Dynamic filtering and sorting
- Authorization at query level (row-level filtering)

## 2. Objective

> Implement DeleteTask, CompleteTask, and ReopenTask commands/endpoints with ownership authorization.

These three commands complete the task lifecycle. DeleteTask soft-deletes or hard-deletes the task (owner only). CompleteTask transitions status from Todo/InProgress → Done and sets `CompletedAt`. ReopenTask reverses Done → Todo and clears `CompletedAt`. Status transitions enforce valid state changes — you cannot complete an already-done task, and you cannot reopen a task that isn't done. This task teaches state machine patterns, domain invariants, and the full task workflow that users interact with daily.

## 4. Technical Guidance

**Conventions for this module:**
- Commands: `DeleteTaskCommand`, `CompleteTaskCommand`, `ReopenTaskCommand` (all take `Id`)
- Delete: ownership check, remove from DbContext or set `IsDeleted` flag (soft delete preferred for Notes/Projects pattern in Phase 04+; can use hard delete for tasks in Phase 03)
- Complete: load task, verify not already Done, set `Status = Done`, set `CompletedAt = DateTime.UtcNow`
- Reopen: load task, verify currently Done, set `Status = InProgress` (or `Todo`), set `CompletedAt = null`
- Validators: `CompleteTaskCommandValidator` checks task exists and status is not Done; `ReopenTaskCommandValidator` checks task exists and status is Done
- Endpoints: `DELETE /api/v1/tasks/{id}`, `POST /api/v1/tasks/{id}/complete`, `POST /api/v1/tasks/{id}/reopen`
- Responses: Delete → 204 No Content; Complete/Reopen → 200 with updated TaskDto

**Concepts practiced in this task:**
- Domain state transitions (status workflow)
- Domain invariants (cannot complete twice)
- Soft delete vs hard delete tradeoffs

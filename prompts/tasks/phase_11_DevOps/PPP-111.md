## 2. Objective

> Add linting and format checks to CI: ESLint (frontend), dotnet format --verify-no-changes (backend).

Linting and format checks in CI enforce code quality mechanically — no human review needed for style issues. The ESLint step runs `npm run lint` (or `eslint src/ --ext .ts,.tsx`) with `--max-warnings 0`, failing the build on any warning. The .NET format step runs `dotnet format --verify-no-changes`, failing if any file needs formatting (spacing, using order, naming). Both gates run in parallel after the build succeeds. This task teaches quality gates in CI, style enforcement automation, and the L3 maturity target for linting.

## 4. Technical Guidance

**Conventions for this module:**
- Add to existing `.github/workflows/ci.yml`:
  - `lint-frontend`: run after build-frontend; `npm run lint` (configured in PPP-006)
  - `lint-backend`: run after build-backend; `dotnet format backend/ --verify-no-changes`
- ESLint config from `.eslintrc.cjs` — must exit with non-zero if warnings/errors
- `dotnet format` requires `.editorconfig` from PPP-001
- Both lint jobs fail the workflow if issues found

**Concepts practiced in this task:**
- Linting as a CI quality gate
- Format enforcement automation
- Fail-fast CI pipeline design

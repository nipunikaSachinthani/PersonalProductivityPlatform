## 2. Objective

> Implement AssignTaskToProject command, project-scoped task list endpoint, and project-level task authorization.

Tasks can be assigned to a project (setting the `ProjectId` FK) or unassigned (setting it to null). Only project members with Owner/Member roles can assign tasks to that project. A new endpoint `GET /api/v1/projects/{id}/tasks` returns only tasks belonging to that project, with the same pagination/filtering as the main task list but scoped to project members. This task bridges the Tasks and Projects modules, teaching cross-module integration, scoped resource endpoints, and authorization that spans two domain boundaries.

## 4. Technical Guidance

**Conventions for this module:**
- Command: `AssignTaskToProjectCommand` (TaskId, ProjectId?) — null ProjectId means unassign
- Handler: verify current user is Owner/Member of project (if assigning); verify task ownership; set `TaskItem.ProjectId`; save
- Query: `GetProjectTasksQuery` (ProjectId, page, pageSize, status, priority, sortBy, sortOrder) — verify membership, return paginated tasks for project
- Endpoints: `PUT /api/v1/tasks/{id}/project` (body: `{ projectId }`), `GET /api/v1/projects/{id}/tasks`
- Authorization: project-scoped task endpoint uses membership check instead of ownership check
- Response: standard `ApiResponse` with paginated task list

**Concepts practiced in this task:**
- Cross-module integration
- Scoped resource endpoints
- Dual authorization (task ownership + project membership)
- Nullable FK management

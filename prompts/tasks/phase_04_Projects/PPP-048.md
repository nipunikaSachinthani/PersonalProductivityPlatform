## 2. Objective

> Write Phase 04 light tests: authorization policy test (non-member denied), assign task to project integration test, and member management integration test.

These tests focus on the new authorization complexity introduced in Phase 04 — membership-based access control. The authorization test creates a project with members, then verifies that non-members are denied access (404) to project endpoints. The assign task test verifies the full task-to-project lifecycle: create a project, create a task, assign it, verify it appears in project tasks, reassign. The member management test verifies add/remove flows including the last-owner invariant. These tests ensure the collaboration boundaries are watertight before more features are added.

## 4. Technical Guidance

**Conventions for this module:**
- Integration test 1: `ProjectAuthorizationTests` — create project as User A; try to access as User B (non-member) → 404; add User B as Viewer → can GET but cannot PUT → 403; add User B as Member → can PUT → 200
- Integration test 2: `TaskProjectAssignmentTests` — create project, create task, assign task to project (200), verify task appears in project tasks, unassign (set projectId to null), verify task no longer in project
- Integration test 3: `MemberManagementTests` — add member (201), add same member again (409), remove member (204), remove last owner (400 with error message)
- All use `CustomWebApplicationFactory` with seeded test data or per-test setup

**Concepts practiced in this task:**
- Authorization integration testing
- Multi-user test scenarios
- Domain invariant testing
- Test data isolation

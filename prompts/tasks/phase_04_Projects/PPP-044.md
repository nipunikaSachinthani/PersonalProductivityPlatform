## 2. Objective

> Implement AddMember and RemoveMember commands with authorization (Owner only) and domain invariants.

Member management is an Owner-only operation. Adding a member looks up the user by email, checks they are not already a member, and creates a ProjectMember record with the specified role. Removing a member enforces the domain invariant that a project cannot have zero owners — attempting to remove the last Owner is rejected. The handler also prevents an Owner from removing themselves (the last owner cannot leave without transferring or deleting). This task teaches complex domain invariants, role validation, and multi-step authorization checks.

## 4. Technical Guidance

**Conventions for this module:**
- Commands: `AddProjectMemberCommand` (ProjectId, Email, Role), `RemoveProjectMemberCommand` (ProjectId, UserId)
- Add handler: verify current user is Owner; find target user by email; verify not already a member; create ProjectMember; save
- Remove handler: verify current user is Owner; find the membership; verify not removing the last Owner (count Owner members, must be > 1 or the removed member is not Owner); save
- Validators: Email required/valid; Role must be valid enum
- Endpoints: `POST /api/v1/projects/{id}/members`, `DELETE /api/v1/projects/{id}/members/{userId}`
- Response: Add → 201 with MemberDto; Remove → 204 No Content
- Edge cases: adding self when already a member → 409; user not found → 404; removing last Owner → 400 with descriptive error

**Concepts practiced in this task:**
- Complex domain invariants (cannot remove last owner)
- Multi-entity transactional writes
- Role hierarchy enforcement
- Edge case handling for membership state

## 2. Objective

> Create Projects and ProjectMembers tables with EF Core migration, and add ProjectId FK to Tasks.

This task introduces a new domain module — Projects — and the relational glue that connects it to Users (many-to-many via ProjectMembers) and Tasks (one-to-many). The Projects table stores project metadata (name, description, owner, archive status). ProjectMembers is a join table with a composite key and role enum (Owner/Member/Viewer). The Tasks table gains a nullable `ProjectId` FK, enabling project-scoped task lists. This task teaches many-to-many relationship design, composite key configuration, and schema evolution (altering existing tables via migration).

## 4. Technical Guidance

**Conventions for this module:**
- Entities in `Domain/Entities/`: `Project`, `ProjectMember`
- Project fields: `Id (int)`, `Name (string, max 200)`, `Description (string, max 2000)`, `OwnerUserId (int, FK→Users)`, `CreatedAt (UTC)`, `IsArchived (bool, default false)`
- ProjectMember fields: `ProjectId (int, FK)`, `UserId (int, FK)`, `Role (ProjectRole enum: Owner, Member, Viewer)`, composite PK on (ProjectId, UserId)
- Add `ProjectId? (int, FK→Projects)` to `TaskItem` entity
- EF config: `HasMany(p => p.Members).WithOne().HasForeignKey(pm => pm.ProjectId)`; configure composite key with `HasKey(pm => new { pm.ProjectId, pm.UserId })`
- `ProjectRole` enum in `Domain/Enums/`
- Index on `Projects.OwnerUserId`, `ProjectMembers.ProjectId`, `TaskItem.ProjectId`

**Concepts practiced in this task:**
- Many-to-many relationships via explicit join entity
- Composite primary keys
- Foreign key relationships and cascade behavior
- Schema migration that alters existing tables

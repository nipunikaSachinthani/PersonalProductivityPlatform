## 2. Objective

> Implement UpdateProject and ArchiveProject commands with role-based authorization (Owner/Member only).

Project updates (name, description) and archiving require the user to be an Owner or Member of the project — Viewers are read-only. ArchiveProject soft-deletes the project by setting `IsArchived = true`; archived projects are hidden from default queries but recoverable. The authorization check ensures only project members with sufficient permissions can modify project state. This task teaches role-based resource authorization beyond simple ownership and soft-delete patterns.

## 4. Technical Guidance

**Conventions for this module:**
- Commands: `UpdateProjectCommand` (Id, Name, Description), `ArchiveProjectCommand` (Id)
- Authorization: load project with members, find current user's membership, verify `Role == Owner || Role == Member`; Viewer or non-member → 403/404
- Update handler: verify member role ≥ Member, update Name/Description, save
- Archive handler: verify member role ≥ Member, set `IsArchived = true`, save
- Validator: Name required 1-200 (same as create)
- Endpoints: `PUT /api/v1/projects/{id}`, `POST /api/v1/projects/{id}/archive`
- Responses: Update → 200 with ProjectDto; Archive → 200 with ProjectDto (archived state)

**Concepts practiced in this task:**
- Role-based resource authorization (Owner/Member vs Viewer)
- Soft delete pattern
- Domain state management (IsArchived flag)

## 2. Objective

> Configure ESLint, Prettier, and EditorConfig for the frontend project to enforce consistent code style.

Static analysis and formatting are mechanical enforcers of code quality — they catch bugs, style violations, and formatting inconsistencies before code review. ESLint with TypeScript rules catches common React and JS mistakes. Prettier eliminates style debates by auto-formatting on save. EditorConfig ensures basic settings (indentation, charset, line endings) are consistent across all editors. This task targets the L0→L2 maturity jump for static analysis defined in ENGINEERING_MATURITY.md and prevents style drift as more code is added.

## 4. Technical Guidance

**Conventions for this module:**
- ESLint: extends `eslint:recommended`, `plugin:@typescript-eslint/recommended`, `plugin:react-hooks/recommended`
- Prettier: `singleQuote: true`, `semi: true`, `trailingComma: 'es5'`, `printWidth: 100`, `tabWidth: 2`
- EditorConfig: 2-space for `.ts`, `.tsx`, `.js`, `.json`; 4-space for `.cs`; `end_of_line = lf`, `charset = utf-8`
- `.prettierignore`: `node_modules/`, `dist/`, `coverage/`, `build/`
- Add npm scripts: `"lint": "eslint src/ --ext .ts,.tsx"`, `"format": "prettier --write src/"`, `"format:check": "prettier --check src/"`
- Optional (stretch): Husky pre-commit hook running `lint-staged`

**Concepts practiced in this task:**
- Static analysis as quality gate
- Automatic code formatting
- Cross-editor configuration standards

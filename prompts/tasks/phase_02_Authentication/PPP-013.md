## 2. Objective

> Implement refresh token rotation flow and logout: `RefreshToken` and `Logout` commands with their API endpoints.

Refresh tokens enable silent session renewal — when the short-lived access token (15 min) expires, the frontend sends the refresh token to get a new access token without requiring the user to log in again. For security, each refresh rotates the token: the old one is revoked, and a new one is issued (rotation prevents token replay attacks). Logout revokes all refresh tokens for the user, effectively ending the session. This task teaches token lifecycle management, rotation as a security pattern, and session termination semantics.

## 4. Technical Guidance

**Conventions for this module:**
- Commands: `RefreshTokenCommand` (takes refresh token string), `LogoutCommand` (takes refresh token string; authenticated user for revocation)
- Refresh handler: validate token exists and is not revoked/expired, revoke old token, issue new access + refresh token pair
- Logout handler: revoke the provided refresh token; optionally revoke all user's tokens
- Token storage: refresh token stored as SHA-256 hash in DB (only the hash, not the raw token)
- Response for refresh: same shape as login (`{ accessToken, refreshToken, expiresAt }`)
- Response for logout: 204 No Content
- Endpoints: `POST /api/v1/auth/refresh-token`, `POST /api/v1/auth/logout`

**Concepts practiced in this task:**
- Refresh token rotation (security pattern)
- Token lifecycle management
- Session termination
- One-way hashing for stored tokens

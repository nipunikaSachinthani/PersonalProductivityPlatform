## 2. Objective

> Implement user registration: `RegisterUser` MediatR command, FluentValidation, and `POST /api/v1/auth/register` endpoint.

Registration is the entry point for all users. The command handler validates input (email format, password strength, name length), checks for duplicate emails, hashes the password using ASP.NET Core's `PasswordHasher<T>`, creates the User entity, assigns the default "User" role, and returns a success response. The DTO and validator follow the Clean Architecture pattern where validation happens at the Application layer boundary. This task teaches the full MediatR command pipeline, password security, duplicate detection, and structured API responses.

## 4. Technical Guidance

**Conventions for this module:**
- Command: `RegisterUserCommand` with `Email`, `Password`, `FirstName`, `LastName` in Application layer
- Validator: `RegisterUserCommandValidator` using FluentValidation — email not empty/valid, password min 8 chars with complexity, first/last name required
- Handler: `RegisterUserCommandHandler` — check unique email, hash password, save user, assign "User" role
- Password hashing: `PasswordHasher<User>()` — store only the hash
- Endpoint: `AuthController.Register()` → `POST /api/v1/auth/register`
- Response: `ApiResponse<UserDto>` with 201 Created; duplicate email returns 409 Conflict with error detail
- `UserDto` mapping (no password hash in response)

**Concepts practiced in this task:**
- MediatR command pipeline
- FluentValidation at API boundary
- Password hashing best practices
- Duplicate detection and conflict responses
- DTO pattern (separate from entity)

## 2. Objective

> Implement MediatR NotificationHandlers that consume domain events and create Notification records in the database.

These handlers are the "consumer" side of the event pipeline. `TaskCompletedNotificationHandler` receives `TaskCompletedEvent` and creates a notification for the task owner. `TaskAssignedNotificationHandler` receives `TaskAssignedEvent` and creates notifications for each assigned project member. `DueDateApproachingNotificationHandler` (used by the background job in PPP-073) receives `DueDateApproachingEvent` and creates reminder notifications. Each handler is idempotent â€” running the same event twice should not create duplicate notifications. This task teaches event consumption, the handler pattern, and idempotent side effects.

## 4. Technical Guidance

**Conventions for this module:**
- Handlers in `Application/Notifications/Handlers/`: implement `INotificationHandler<TEvent>`
- `TaskCompletedNotificationHandler`: on `TaskCompletedEvent`, create Notification with Type="TaskCompleted", Title="Task Completed", Message=$"'{event.TaskTitle}' has been completed"
- `TaskAssignedNotificationHandler`: on `TaskAssignedEvent`, create Notification for each assigned user (skip self-assignment)
- `DueDateApproachingNotificationHandler`: on `DueDateApproachingEvent`, create Notification with Type="DueDateReminder", Title="Due Date Approaching", Message=f"'{event.TaskTitle}' is due {event.DueDate:MMM dd}"
- Idempotency check: before creating, check if a notification with same Type, UserId, RelatedEntityType, RelatedEntityId already exists within last hour; skip if so
- Handler must not throw â€” catch exceptions, log at ERROR level with full context (event type, related entity IDs â€” no PII), and exit gracefully. Emit a custom metric (`notification.handler.failed`) so silent pipeline degradation is observable. The pipeline must continue processing subsequent handlers after a single handler fails.

**Concepts practiced in this task:**
- Event consumption with MediatR notification handlers
- Idempotent event handling
- Side-effect isolation
- Defensive handler design (no-throw)

## 2. Objective

> Optional capstone: Extract the Notification Service into a separate microservice behind a message bus (Azure Service Bus or RabbitMQ).

This is the capstone architecture exercise. The Notification handlers and background jobs are extracted from the monolith into a separate ASP.NET project (`NotificationService`). A message bus (Azure Service Bus locally via connection string, or RabbitMQ via Docker) replaces the in-process MediatR events. The Task service publishes `TaskCompletedEvent` and `TaskAssignedEvent` to the bus; the Notification service consumes them and creates notifications in its own database. This mirrors enterprise microservices patterns without over-engineering — the Notification Service is the smallest, most event-driven module. This task teaches service extraction, message bus integration, and the strangler fig pattern.

## 4. Technical Guidance

**Conventions for this module:**
- New project: `ProductivityPlatform.NotificationService` — minimal API or Web API project
- Message bus: choose RabbitMQ (local Docker) for learning or Azure Service Bus (cloud-native)
  - RabbitMQ: `docker run -d --name rabbitmq -p 5672:5672 -p 15672:15672 rabbitmq:3-management`
  - Azure Service Bus: use connection string from Azure; emulate with Azurite (limited support)
- Publisher (Task service): inject `IMessageBus`, publish events as JSON messages to queue/topic
- Consumer (Notification service): `BackgroundService` that listens to queue, deserializes events, saves to DB
- Message contracts: shared NuGet package or linked files with `TaskCompletedIntegrationEvent`, `TaskAssignedIntegrationEvent`
- Notification database: shared SQL Server (same or separate — separate for full microservice pattern)
- API: Notification service exposes `GET /api/notifications` and read/mark endpoints directly

**Concepts practiced in this task:**
- Microservices extraction (strangler fig)
- Message bus integration
- Event-driven architecture at scale
- Service boundaries and data ownership

## 2. Objective

> Create Attachments table with EF Core migration and configure Azure Blob Storage (Azurite emulator locally).

The Attachments table stores file metadata — the actual files live in Azure Blob Storage. In development, Azurite (a local Azure Storage emulator) provides the same API without cloud costs. The schema stores the file name, blob URL, content type, size, uploader, and task association. This task teaches cloud storage modeling (metadata in DB, files in blob), local emulation for development, and the separation of concerns between file storage and relational data.

## 4. Technical Guidance

**Conventions for this module:**
- Entity: `Attachment` in `Domain/Entities/`
- Fields: `Id (int)`, `TaskId (int, FK→Tasks)`, `FileName (string, max 255)`, `BlobUrl (string)`, `ContentType (string, max 100)`, `SizeBytes (long)`, `UploadedByUserId (int, FK→Users)`, `UploadedAt (UTC)`
- Table: `Attachments`; index on `TaskId`
- Azure Blob: install `Azure.Storage.Blobs` NuGet package
- Azurite: run via Docker (`docker run -p 10000:10000 mcr.microsoft.com/azure-storage/azurite`) or npm (`npx azurite`)
- Connection string for Azurite: `DefaultEndpointsProtocol=http;AccountName=devstoreaccount1;AccountKey=Eby8vdM02xNOcqFlqUwJPLlmEtlCDXJ1OUzFT50uSRZ6IFsuFq2UVErCz4I6tq/K1SZFPTOtr/KBHBeksoGMGw==;BlobEndpoint=http://127.0.0.1:10000/devstoreaccount1;`
- Store connection string in User Secrets
- Create `IBlobStorageService` interface in Application, implementation in Infrastructure

**Concepts practiced in this task:**
- Cloud storage modeling (metadata + blob separation)
- Local emulation (Azurite)
- Blob storage abstraction (interface/implementation)
